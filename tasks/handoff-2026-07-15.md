## Stage 27 CLI

```text
--representation snapshot-json
--envelope <validated-project-relative-path>
[--task-id <task-YYYYMMDD-NNN>] [--request-id <req-YYYYMMDD-NNN>]
```

- filter 至少一个；
- 双参数使用 AND；
- filter 必须与显式 envelope 同时使用；
- 每个 filter flag 最多一次，duplicate 输入结构化失败；
- canonical ASCII exact match，不 trim、不折叠大小写；
- 不接受 list、wildcard、regex、通用 query、sort、page 或 export。

## 匹配与关系闭包

filter 只作用于完整验证后的 runs/approvals/artifacts summaries：

- task-only：先选择 `run.task_id`，再以 selected request ids 包含关联 approval/artifact；
- request-only：三个 section 均按 `request_id` exact match；
- task+request：run 必须同时匹配，再以 selected request set 过滤关联 section；
- 二者不一致或无匹配：返回 `ready`、`matched=false` 的合法空视图。

这使 `adapter_response` 即使安全摘要中 `task_id=""`，仍可通过 request→task 关系闭包进入正确 task view。

## v1 / v2 / v3 兼容

| 输入 | 输出 |
|:---|:---|
| 无 envelope、无 filter | `control-plane/codex-desktop-snapshot-read/v1` |
| envelope、无 filter | `control-plane/codex-desktop-snapshot-read/v2` |
| envelope + task/request filter | `control-plane/codex-desktop-snapshot-read/v3` |
| filter、无 envelope | `validation_failed` / `filter-envelope-required` |
任何显式提供 task/request filter 的请求（包括 filter/envelope 输入门禁失败）均使用 v3 wrapper；未请求 filter 的既有 v1/v2 失败口径不变。


v3 reader id：`codex-desktop-filtered-envelope-snapshot-json-reader/v3`。

## Identity

```text
filter_id = sha256(canonical(filter))
view_id = sha256(canonical(filtered_payload_without_view_id))
```

输出前独立重算并验证：

- canonical filter shape 与 filter id；
- base snapshot id / scope id 形状；
- filtered payload canonical view hash。

representation 分别输出 `base_snapshot_id` 与 `view_id`，不会拿 base snapshot id 冒充 filtered view identity。

## 生命周期与固定执行链

```text
created → scoping → producing → validating → reading → filtering → ready → closed
```

固定 child argv 仍只有：

```text
handoff --envelope <validated-path> --json
reference consumer(stdin=handoff bytes)
snapshot --envelope <validated-path> --json
```

filter 不传给 child process，也不写入 descriptor。

## Stage 28 条件启动判定

Stage 28 只有在用户明确要求继续推进，或出现一个具体宿主需要独立消费 filtered v3 时才启动。启动前必须能回答：

1. 宿主是谁，为什么不能直接信任 reader stdout；
2. consumer 接收的是完整 v3 reader result，还是只接收 filtered payload；
3. consumer 需要验证哪些 cross-field identity，哪些只能检查形状而不能重算；
4. `ready`、`validation_failed`、`blocked`、`error` 如何映射为宿主状态和退出码；
5. 宿主只展示哪些安全 summaries，是否需要比 reader 更窄的输出；
6. 输入和输出上限是否继续为 1 MiB，是否保持 stdin-only / stdout-only；
7. 如何证明 consumer 不执行 argv、不读取 envelope、不访问项目文件、不保存 view。

若没有具体消费者或这些问题无法以有界契约回答，继续维持 Stage 27 冻结状态，不创建“通用 consumer”。

## Stage 28 首轮设计审计范围

条件满足后，第一拍只做 design gate，建议创建新的活跃事实源：

```text
docs/88-filtered-snapshot-host-consumer-validation-gate.md
```

该文档应冻结：

- 输入：单个 bounded UTF-8 JSON document，经 stdin 提供；
- 接受版本：仅显式 allowlist 的 v3 reader result / filtered payload schema；
- wrapper 校验：reader id、status、lifecycle、source、representation 与 guarantees；
- identity 关联：wrapper/source/payload 的 `filter_id`、`scope_id`、`base_snapshot_id`、`view_id` 必须一致；
- 可重算 identity：canonical filter id 与 filtered payload view id；
- 不可伪造事实：consumer 不能声称重算 Stage 24 base snapshot id，除非输入明确携带并允许验证 base payload；
- safe-summary 边界：只接受 runs/approvals/artifacts/reports，不接受 project/registry/raw envelope 扩展；
- 生命周期和退出码：一次性、无自动重试、失败不降级为 ready；
- 输出：比输入更小的确定性 validation result，不回显整份 payload；
- 执行边界：不 import producer builder、不执行 descriptor/child argv、不读文件、不访问网络、不写入。

Stage 28 design gate 不应直接实现宿主 bridge、长期进程、缓存、export 或 UI。

## Stage 28 候选 TDD 矩阵

进入实现阶段之前，至少冻结以下用例：

1. valid filtered v3 result；
2. unsupported schema / reader id；
3. 非 ready 或 lifecycle 未 closed；
4. unsafe guarantees；
5. wrapper 与 payload filter/scope/base/view identity 不一致；
6. canonical filter id 重算失败；
7. filtered view hash 重算失败；
8. duplicate JSON key、非 UTF-8、空输入、非 object、超过上限；
9. payload 出现 project/registry/raw 字段；
10. finding/message 不回显 task/request 之外的 raw value；
11. deterministic output 与 no-write/no-network/no-execute；
12. 真实 reader stdout → 独立 consumer stdin 的本地 one-shot smoke。

是否实现新的 consumer 工具，应由 Stage 28 design gate 结论决定；不得预设必须实现。
