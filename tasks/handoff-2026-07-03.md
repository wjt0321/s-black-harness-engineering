## 下一阶段候选路线

### 路线 A — 打 v0.1 标签

适合短收口。

要做：

1. 本地确认工作树干净。
2. 跑测试、doctor、public scan。
3. 创建 tag：`v0.1.0-planning-poc`。
4. 经用户确认后推送 tag。
5. 可补 GitHub Release 草稿。

### 路线 B — 扩展 check action

适合继续增强 policy 能力。

要做：

1. 让 `check action` 更完整读取 `publish_rules`。
2. 支持 `required_checks` 输出。
3. 支持 completion rule 的基础判断。
4. 增加测试。

注意：仍然只做 preflight，不执行外部动作。

### 路线 C — 真实 task ledger 只读查询

适合让 `task status` / `task events` 真正可用。

要做：

1. 新增 `tasks/tasks.jsonl` 示例或本地文件约定。
2. 实现 `task status <task-id>`。
3. 实现 `task events <task-id>`。
4. 不写入，只查询。

### 路线 D — 项目工程化

适合让外部用户更容易跑。

要做：

1. 增加 `pyproject.toml`。
2. 定义 console script，但不要求全局安装。
3. 补最小依赖说明。
4. 考虑 GitHub Actions：跑 tests + doctor。

### 路线 E — Adapter 执行层设计二期

适合继续大设计。

要做：

1. 设计真实 adapter execution envelope。
2. 定义 approval token / approval ref 的非持久化模型。
3. 明确执行层如何写 event。
4. 仍不接入真实外部系统。

## 建议下一步

建议下一次从 **路线 A：打 v0.1 标签** 开始。

理由：

- 当前已经有阶段性成果。
- 先冻结一个可回退点。
- 后续扩展 check action 或 task ledger 时，不会把 v0.1 的稳定边界冲掉。

如果继续开发，建议顺序：

1. v0.1 tag
2. task status / task events 只读查询
3. check action publish_rules 扩展
4. pyproject + GitHub Actions
